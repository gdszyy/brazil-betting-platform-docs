## 1. 概述

本方案基于总体设计文档中的指导原则，为巴西墨西哥体育博彩平台的【分享系统】提供详细的技术实现方案。方案旨在构建一个集成了多样化分享方式、精确归因追踪、高效短链服务及动态海报生成的综合性社交裂变系统。

核心需求涵盖：

1.  **四种分享方式**: 系统分享 (Web Share API)、三方分享 (Facebook/TikTok SDK)、图片分享 (二维码海报)、网页链接分享。
2.  **归因追踪**: 通过链接参数（`sharer_id`, `campaign_id`, `content_id`, `timestamp`）实现用户来源追踪，并支持设置归因有效期。
3.  **短链服务**: 将携带参数的长链接转换为短链，要求支持高并发访问和快速重定向。
4.  **分享海报**: 后端根据请求动态生成包含活动信息和专属二维码的分享海报。

本文档将依次详细阐述数据库设计、API接口、归因流程、短链服务技术方案和分享海报生成方案。

## 2. 数据库表结构设计

为了支持分享系统的功能，我们设计了以下三张核心数据表：分享记录表 (`share_records`)、分享归因表 (`share_attributions`) 和短链接映射表 (`short_links`)。

### 2.1. 分享记录表 (`share_records`)

此表用于记录每一次由用户发起的分享行为。

| 字段名 | 数据类型 | 长度/默认值 | 是否可空 | 主键/索引 | 字段说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | - | No | PK | 唯一ID，自增 |
| `share_id` | `VARCHAR` | 64 | No | UK | 分享唯一标识，UUID |
| `user_id` | `BIGINT` | - | No | IX | 分享者用户ID |
| `campaign_id` | `VARCHAR` | 64 | Yes | IX | 关联的活动ID |
| `content_id` | `VARCHAR` | 128 | Yes | - | 分享的具体内容ID（如注单号） |
| `share_type` | `VARCHAR` | 20 | No | - | 分享方式 (system, third_party, image, link) |
| `platform` | `VARCHAR` | 30 | Yes | - | 分享平台 (facebook, tiktok, whatsapp等) |
| `long_url` | `TEXT` | - | No | - | 原始长链接 |
| `short_url` | `VARCHAR` | 100 | No | IX | 生成的短链接 |
| `created_at` | `DATETIME` | `CURRENT_TIMESTAMP` | No | - | 创建时间 |

### 2.2. 分享归因表 (`share_attributions`)

此表用于记录通过分享链接带来的新用户注册或转化事件，建立分享者与新用户的归因关系。

| 字段名 | 数据类型 | 长度/默认值 | 是否可空 | 主键/索引 | 字段说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | - | No | PK | 唯一ID，自增 |
| `share_id` | `VARCHAR` | 64 | No | IX | 关联的分享ID (`share_records.share_id`) |
| `sharer_id` | `BIGINT` | - | No | IX | 分享者用户ID |
| `new_user_id` | `BIGINT` | - | No | UK | 被邀请的新用户ID |
| `attribution_type` | `VARCHAR` | 20 | No | - | 归因类型 (e.g., 'register', 'first_deposit') |
| `campaign_id` | `VARCHAR` | 64 | Yes | IX | 关联的活动ID |
| `status` | `VARCHAR` | 20 | `pending` | - | 归因状态 (pending, confirmed, expired) |
| `expires_at` | `DATETIME` | - | No | - | 归因有效期截止时间 |
| `created_at` | `DATETIME` | `CURRENT_TIMESTAMP` | No | - | 创建时间 |
| `updated_at` | `DATETIME` | `ON UPDATE` | No | - | 更新时间 |

### 2.3. 短链接映射表 (`short_links`)

此表用于存储短链接与原始长链接的映射关系，是短链服务的核心数据结构。

| 字段名 | 数据类型 | 长度/默认值 | 是否可空 | 主键/索引 | 字段说明 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| `id` | `BIGINT` | - | No | PK | 唯一ID，自增 |
| `short_code` | `VARCHAR` | 10 | No | UK | 短链接代码（URL中的唯一部分） |
| `long_url` | `TEXT` | - | No | - | 原始长链接 |
| `total_clicks` | `INT` | 0 | No | - | 总点击次数 |
| `created_at` | `DATETIME` | `CURRENT_TIMESTAMP` | No | - | 创建时间 |
| `last_accessed_at` | `DATETIME` | Null | Yes | - | 最后访问时间 |

## 3. 核心API接口定义 (RESTful)

以下是分享系统的核心API接口定义。

### 3.1. 创建分享链接

- **Endpoint**: `POST /api/v1/shares`
- **Description**: 用户发起分享时，客户端调用此接口为目标内容生成一个可追踪的分享链接（已包含短链）。
- **Request Body**:
  ```json
  {
    "campaign_id": "cmp_summer_promo_2026",
    "content_id": "bet_slip_123456789",
    "share_type": "image",
    "platform": "whatsapp",
    "target_url": "https://www.xp-bet.com/event/12345"
  }
  ```
- **Success Response (200 OK)**:
  ```json
  {
    "share_id": "shr_a1b2c3d4e5f6",
    "short_url": "https://sh.rt/xY8zPq",
    "poster_url": "https://cdn.xp-bet.com/posters/shr_a1b2c3d4e5f6.png" // 仅在share_type为image时返回
  }
  ```

### 3.2. 短链接重定向

- **Endpoint**: `GET /s/{short_code}`
- **Description**: 用户访问短链接时，系统通过此接口完成重定向，并记录点击。这是短链服务的入口。
- **Action**:
  1.  根据 `{short_code}` 在 `short_links` 表或其缓存中查找 `long_url`。
  2.  如果找到，异步增加 `total_clicks` 计数。
  3.  返回 `302 Found` 状态码，并在 `Location` 头中设置原始 `long_url`。
  4.  如果未找到，返回 `404 Not Found`。

### 3.3. 记录归因事件 (内部服务调用)

- **Endpoint**: `POST /api/v1/internal/attributions`
- **Description**: 当新用户完成关键转化行为（如注册）时，由用户系统或支付系统在后端调用此接口，记录归因关系。
- **Request Body**:
  ```json
  {
    "new_user_id": 98765,
    "share_id": "shr_a1b2c3d4e5f6", // 从用户会话或设备指纹中获取
    "attribution_type": "register"
  }
  ```
- **Success Response (200 OK)**:
  ```json
  {
    "status": "success",
    "message": "Attribution recorded successfully."
  }
  ```

## 4. 归因追踪完整流程设计

归因追踪是分享系统的核心，确保分享者的贡献能被准确记录并获得奖励。完整流程如下：

1.  **步骤一：生成分享链接**
    - 用户A（分享者）在客户端点击分享按钮。
    - 客户端调用 `POST /api/v1/shares` 接口，请求中包含 `campaign_id`、`target_url` 等信息。
    - 后端生成唯一的 `share_id`，并将 `sharer_id` (用户A的ID), `campaign_id`, `content_id`, `timestamp` 等参数组合成一个原始长链接 `long_url`。
    - 后端调用短链服务，为 `long_url` 生成一个短链接 `short_url`。
    - 后端将 `share_id`, `long_url`, `short_url` 等信息存入 `share_records` 表。
    - 如果分享类型是图片，后端还会触发海报生成任务，并将海报URL返回。
    - 接口返回 `short_url` 给客户端，用户A将其分享出去。

2.  **步骤二：新用户访问与追踪**
    - 新用户B（被邀请者）点击了用户A分享的 `short_url`。
    - 浏览器请求 `GET /s/{short_code}`。
    - 短链服务将请求重定向到原始的 `long_url`。
    - 此时，用户B的浏览器地址栏变为 `https://www.xp-bet.com/event/12345?share_id=shr_a1b2c3d4e5f6&...`。
    - 平台前端JS获取URL中的 `share_id` 参数，并将其存储在用户B的 **Cookie** 或 **LocalStorage** 中，同时设置一个**归因有效期**（例如7天）。这个 `share_id` 就像一个追踪信标。

3.  **步骤三：转化与归因确认**
    - 用户B在归因有效期内（例如第3天）决定注册成为平台新用户。
    - 在注册流程中，前端将存储的 `share_id` 随注册信息一同提交给后端。
    - 后端用户服务在创建新用户账户后，检测到附带的 `share_id`，立即调用 `POST /api/v1/internal/attributions` 接口。
    - 归因服务收到请求，在 `share_attributions` 表中创建一条记录，关联 `share_id`、分享者ID (`sharer_id`) 和新用户ID (`new_user_id`)，并设置归因状态为 `confirmed` 和有效期 `expires_at`。

4.  **步骤四：奖励发放**
    - 一个独立的定时任务或事件监听器，在检测到 `share_attributions` 表中出现新的 `confirmed` 记录时，触发奖励发放流程。
    - 奖励发放服务根据 `campaign_id` 和归因记录，向分享者用户A的账户发放预设的奖励（如彩金、抽奖券等）。

## 5. 短链服务技术方案

短链服务需要满足高并发、低延迟的核心要求。

- **技术栈**: **Redis + Go/Node.js (Lwan/Nginx)**

- **方案详述**:
    1.  **短码生成算法**: 使用 **Base62** 编码。当需要生成新的短码时，从数据库获取一个自增的唯一ID（例如 `short_links` 表的 `id`），然后将这个ID从10进制转换为62进制（`[0-9a-zA-Z]`）。例如，ID `10000` 转换为Base62是 `gN`。可以预先生成一批ID并打乱，以避免短码被顺序预测。

    2.  **存储与缓存**: **Redis** 作为主存储和缓存层。使用 `HASH` 数据结构存储 `short_code` 到 `long_url` 的映射。`HSET shortlinks {short_code} {long_url}`。Redis的内存访问特性可以提供微秒级的查询速度，完全满足低延迟要求。

    3.  **重定向服务**: 使用高性能的Web服务器如 **Nginx + Lua (OpenResty)** 或一个轻量级的 **Go/Node.js** 应用。服务逻辑极其简单：接收请求，从URL中提取 `short_code`，在Redis中查询对应的 `long_url`，然后返回302重定向。这个服务可以水平扩展以应对极高的并发量。

    4.  **数据持久化与一致性**: Redis开启AOF或RDB持久化，确保数据在服务重启后不丢失。同时，可以将Redis中的数据定期或实时同步到后端的MySQL/PostgreSQL数据库（`short_links`表）中，作为冷备份和数据分析源。

    5.  **点击统计**: 重定向服务在返回302之前，可以向一个消息队列（如Kafka或RabbitMQ）发送一条点击事件消息，包含 `short_code`。一个独立的后台消费者服务负责处理这些消息，更新Redis和数据库中的点击计数值，从而避免阻塞重定向的关键路径。

## 6. 分享海报生成方案

分享海报需要在后端动态生成，以确保每个用户的二维码都是独一无二的。

- **技术栈**: **Node.js + Puppeteer / Python + Pillow**

- **方案详述 (以Node.js + Puppeteer为例)**:
    1.  **海报模板**: 使用 **HTML 和 CSS** 创建海报的视觉模板。模板中包含占位符，用于动态填充用户信息（如头像、昵称）、活动文案以及最重要的二维码图片。
        ```html
        <div class="poster-container">
          <img src="{{background_image}}" class="background">
          <img src="{{user_avatar}}" class="avatar">
          <p class="nickname">{{user_nickname}}</p>
          <h1 class="title">{{campaign_title}}</h1>
          <img src="{{qrcode_image_data_uri}}" class="qrcode">
        </div>
        ```

    2.  **二维码生成**: 当 `POST /api/v1/shares` 接口被调用时，后端已经生成了唯一的 `short_url`。使用一个二维码生成库（如 `qrcode`），将这个 `short_url` 转换为一个二维码图片，可以直接生成 **Data URI** 格式的字符串。

    3.  **动态渲染**: 使用 **Puppeteer**（一个无头浏览器库），在后端启动一个隐藏的Chrome浏览器实例。
        - 加载HTML海报模板。
        - 将用户头像、昵称、活动文案以及生成的二维码Data URI等动态数据注入到HTML模板中。
        - Puppeteer渲染这个HTML页面。

    4.  **生成图片并上传**: Puppeteer可以将渲染好的页面内容截图，保存为PNG或JPEG格式的图片文件。
        - 将生成的图片上传到CDN（如AWS S3, Cloudflare R2），获取一个公开可访问的URL (`poster_url`)。
        - 将此 `poster_url` 返回给客户端。

    5.  **性能优化**: Puppeteer实例可以作为常驻服务运行，通过对象池管理浏览器页面，避免频繁创建和销毁进程带来的开销。对于相同的活动模板，可以缓存背景图等静态资源。

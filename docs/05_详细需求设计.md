# 巴西体育博彩平台 - 详细需求设计

**版本:** 1.0
**日期:** 2026-01-12

## 1. 引言

本文档在前序的系统需求与架构设计基础上，对核心模块进行详细的需求设计。本文档采用用户故事 (User Story) 的形式来描述需求，并为每个用户故事提供明确的验收标准 (Acceptance Criteria)，旨在为开发和测试团队提供清晰、可执行的需求说明。

本文档将首先聚焦于 **P0 核心** 级别的模块，包括：

- **用户系统**
- **财务系统**
- **体育注单系统**

## 2. 用户系统 (User System)

### 2.1. 模块目标

为用户提供安全、流畅的注册、登录和账户管理体验，并为平台提供可靠的用户身份验证 (KYC) 和管理能力。

### 2.2. 用户故事与验收标准

#### Epic: 用户注册与登录

> **作为** 一名新用户, 
> **我想要** 通过多种方式（手机号、第三方社交账户）快速注册和登录, 
> **以便于** 我能尽快开始使用平台。

**用户故事 1: 手机号注册**
- **功能点**: `登录注册流程`
- **验收标准**:
  1. 用户输入手机号后，系统能发送短信验证码。
  2. 验证码在5分钟内有效，且只能使用一次。
  3. 用户输入正确的验证码和密码后，系统创建新账户并自动登录。
  4. 系统需校验手机号在巴西的格式是否正确。
  5. 如果手机号已被注册，系统应提示用户直接登录。

**用户故事 2: 第三方账户登录/注册**
- **功能点**: `登录注册流程`
- **验收标准**:
  1. 用户可以选择通过 Google 或 Facebook 账户进行授权。
  2. 首次使用第三方账户授权时，系统自动创建一个新账户并绑定。
  3. 如果该第三方账户已绑定现有账户，系统直接登录。
  4. 成功授权后，用户应跳转回平台并处于登录状态。

**用户故事 3: 用户登录**
- **功能点**: `登录注册流程`
- **验收标准**:
  1. 用户可以使用"手机号+密码"的方式登录。
  2. 用户可以使用"手机号+验证码"的方式登录。
  3. 系统提供"忘记密码"功能，允许用户通过手机验证码重置密码。
  4. 登录失败时，系统应给出明确的错误提示（如"密码错误"或"账户不存在"）。

#### Epic: KYC 与身份验证

> **作为** 一名平台管理员, 
> **我想要** 验证用户的身份信息 (KYC), 
> **以便于** 遵守法规并防止欺诈。

**用户故事 4: KYC流程**
- **功能点**: `KYC流程`
- **验收标准**:
  1. 用户在首次提款前，必须完成KYC认证。
  2. KYC流程要求用户提交身份证明（如身份证、驾照）的正反面照片和手持证件自拍。
  3. 后台管理员可以审核用户提交的KYC材料，并标记为"通过"、"拒绝"或"需要补充材料"。
  4. 审核结果通过站内信或短信通知用户。

#### Epic: 账户管理

> **作为** 一名注册用户, 
> **我想要** 管理我的个人信息和账户安全设置, 
> **以便于** 保护我的账户安全和隐私。

**用户故事 5: 用户信息编辑**
- **功能点**: `用户信息编辑`
- **验收标准**:
  1. 用户可以修改自己的昵称、头像。
  2. 用户的手机号作为唯一标识，不可自行修改，需联系客服处理。

**用户故事 6: 用户自我封禁**
- **功能点**: `用户自我封禁`
- **验收标准**:
  1. 用户可以主动选择封禁自己的账户一段时间（如1天、7天、30天或永久）。
  2. 在封禁期间，用户将无法登录和进行任何投注操作。
  3. 永久封禁操作需要用户进行二次确认。

## 3. 财务系统 (Finance System)

### 3.1. 模块目标

提供安全、高效、可靠的资金处理能力，包括多渠道支付、统一的钱包管理以及清晰的财务操作记录，确保资金流转的准确性和安全性。

### 3.2. 用户故事与验收标准

#### Epic: 钱包管理

> **作为** 一名用户, 
> **我想要** 一个统一的钱包来管理我的所有资金，包括现金、奖金和积分, 
> **以便于** 我能清晰地了解我的资产状况。

**用户故事 7: 多资产钱包**
- **功能点**: `钱包管理`
- **验收标准**:
  1. 每个用户拥有一个主钱包，包含多种子钱包：CASH（现金）、Bonus（奖金）、Freespin（免费旋转）、积分等。
  2. 用户可以清晰地看到每种资产的余额。
  3. 不同资产的使用规则（如Bonus的使用限制）应在界面上有明确说明。

#### Epic: 支付处理

> **作为** 一名用户, 
> **我想要** 通过多种支付方式方便地进行充值和提款, 
> **以便于** 我能快速地参与游戏和获取收益。

**用户故事 8: 支付路由与代收代付**
- **功能点**: `支付路由`, `支付代收代付`
- **验收标准**:
  1. 系统支持接入多个支付渠道（如EPAY, STP, OPAY）。
  2. 后台可以根据支付方式、金额、用户等级等配置支付路由规则。
  3. 系统能处理支付渠道的回调通知，并准确更新订单状态和用户余额。
  4. 所有充值和提款请求都有详细的订单记录，包括订单号、金额、状态、渠道等。

**用户故事 9: 财务操作**
- **功能点**: `财务操作`
- **验收标准**:
  1. 后台管理员可以对用户余额进行手动操作（增加/减少），但需要严格的权限控制和操作日志。
  2. 系统支持对用户钱包进行冻结和解冻操作。
  3. 提款请求需要经过后台审核，审核状态（待审核、已批准、已拒绝）对用户可见。

## 4. 体育注单系统 (Sports Betting System)

### 4.1. 模块目标

提供稳定、实时、功能丰富的体育博彩核心功能，从盘口数据对接到投注交易，再到订单结算，确保用户获得流畅的投注体验。

### 4.2. 用户故事与验收标准

#### Epic: 盘口与赛事展示

> **作为** 一名用户, 
> **我想要** 浏览实时的赛事列表和多样的盘口信息, 
> **以便于** 我能找到感兴趣的比赛并进行投注。

**用户故事 10: 盘口信息展示**
- **功能点**: `盘口信息展示`, `C端盘口页面`
- **验收标准**:
  1. 系统能从第三方数据源（UOF/CTRL）实时拉取并展示赛事和盘口数据。
  2. C端页面以清晰的列表和详情页形式展示赛事信息，包括开赛时间、对阵双方、实时比分等。
  3. 赔率变化时，页面上应有明显的上升（绿色）或下降（红色）提示。
  4. 支持按体育项目（足球、篮球等）、联赛和时间进行筛选和排序。

#### Epic: 投注交易

> **作为** 一名用户, 
> **我想要** 快速、准确地完成投注, 
> **以便于** 抓住最佳的投注时机。

**用户故事 11: 投注交易流程**
- **功能点**: `投注交易`
- **验收标准**:
  1. 用户点击盘口赔率后，该选项自动加入投注栏。
  2. 用户在投注栏输入金额后，系统能实时计算出预计可赢金额。
  3. 用户确认投注后，系统在3秒内完成投注请求，并从用户钱包中扣除相应金额。
  4. 投注成功后，用户可以在"我的投注"中看到该笔注单，状态为"未结算"。
  5. 如果投注时赔率发生变化，系统应提示用户确认是否接受新赔率。

#### Epic: 订单结算与冲正

> **作为** 一名用户, 
> **我想要** 在比赛结束后，我的注单能被快速、准确地结算, 
> **以便于** 我能及时收到我的奖金。

**用户故事 12: 订单清结算**
- **功能点**: `订单清结算`
- **验收标准**:
  1. 系统在收到第三方（UOF）的赛果后，自动对相关注单进行结算。
  2. 结算完成后，如果用户中奖，奖金会自动派发到用户的CASH钱包。
  3. 注单状态更新为"已结算-赢"或"已结算-输"。

**用户故事 13: 冲正处理**
- **功能点**: `冲正处理`
- **验收标准**:
  1. 当出现错误结算时（如赛果更正），系统支持对已结算的注单进行冲正操作。
  2. 冲正操作会回滚原始的结算和派奖动作，并根据新的赛果重新结算。
  3. 所有冲正操作都必须有详细的记录，以备审计。
